VERDICT (choose exactly one):
- approve: No blocking issues in files changed by this PR; SonarQube Quality Gate passed OR not present; automated scans do not show new critical findings in PR changes.
- needs_work: Non-blocking or medium issues **in files this PR changed** (style, minor bugs, workflow YAML fixes).
- reject: Critical/high security **in files this PR changed**, major logic errors, breaking API/schema changes, OR SonarQube Quality Gate FAILED due to issues introduced by this PR.
- comment: Use ONLY when you cannot assess (e.g. empty diff, contradictory missing context) — not as a soft approve.

VERDICT BOUNDARIES (critical):
- The "Changed files in this PR" list is authoritative. Judge needs_work/reject ONLY from that list + diff hunks.
- Repo-wide lint/typecheck/test failures (e2e/, src/, missing LICENSE) when those paths are NOT in the changed-files list → repo_health or check_results_analysis only, NEVER needs_work/reject.
- Workflow-only PRs (.github/workflows/* only): approve unless actionlint/security finds issues in those workflow files.
- If the only findings are pre-existing or outside changed files, verdict MUST be approve.
- Scanner/install failures are environmental — note in check_results_analysis; do not reject for those alone.

REQUIRED OUTPUT — exact reasons (mandatory):
- verdict_rationale: string (newline-separated bullets) OR array of strings — NOT an object. For needs_work/reject, EVERY bullet MUST name a **file path from this PR's changed-files list** and the concrete problem. If nothing in changed files is wrong, verdict MUST be approve.
- verdict_reasons: array of {scope: "pr_change"|"pre_existing"|"check_failure_outside_diff", file: "path or empty", reason: "one clear sentence"}
  - Use scope pr_change only for issues in changed files.
  - Use check_failure_outside_diff for failed lint/test in paths not in the PR (e.g. e2e/ types when PR only touches workflows).

AUTOMATED SCAN INTEGRATION:
- Gitleaks findings in PR range → critical; is_pr_change true; file in changed list.
- actionlint on **changed** workflow files → may drive needs_work with exact file:line.
- Failures in e2e/ or app source when PR did not change those paths → verdict_reasons scope check_failure_outside_diff, NOT needs_work.

SONARQUBE:
- Quality Gate FAILED because of this PR → reject with SonarQube detail in verdict_rationale.
- PASSED with clean diff → strong approve signal.

- Reusable workflow uses: block is valid.
- Unpinned actions may be medium in issues[] only if file is in changed list.
